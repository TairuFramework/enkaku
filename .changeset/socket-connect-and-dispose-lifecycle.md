Socket connect and dispose lifecycle.

- `connectSocket(path, options?)` accepts `{ timeoutMs, signal }`, via the new exported `ConnectSocketOptions` type. The connect attempt now times out after **10 seconds by default** — pass `timeoutMs: 0` for the previous unbounded behaviour. An abandoned attempt (timeout or abort) destroys its pending socket, and both the `connect` and `error` listeners are detached once the promise settles (the `error` one previously stayed attached for the socket's whole life).
- The transport stream's writable close callback now awaits `socket.end()`'s flush, bounded by a 2 second grace, so `writer.close()` no longer resolves with bytes still queued. This makes `dispose()` itself slower in the case it used to paper over: the close callback used to be synchronous (`end(); unref()`); it now awaits the flush, up to 2 seconds. Measured at 910 ms in a stalled-then-recovering peer scenario where it used to be ~0.
- **`SocketTransport.dispose()` now destroys the socket** rather than only `unref()`ing it, and does so for every source shape — including a function source, which previously registered no release hook at all. `unref()` left the socket open, so a peer server could wait forever -- either for a transport that was never read from or written to (whose socket `end()` never ran), or for one whose peer still had data queued to an unreading client. A transport built from a function source that was never read from or written to opens no socket, and therefore has none to release. This means the transport now **takes ownership of a caller-supplied `Socket`** — `new SocketTransport({ socket })` will `destroy()` that socket on `dispose()`, even if the transport was never read from or written to.
- `SocketTransportParams` accepts `connectTimeoutMs`, applied when `socket` is a path string. Such a path is now connected **lazily**, on first read/write, rather than eagerly in the constructor.
- `CreateTransportStreamOptions<R>` gains `signal?: AbortSignal`. A `write()` stuck awaiting `'drain'` against a peer that never reads gets no `'drain'`, `'close'`, or `'error'` — nothing to settle it. `SocketTransport` now passes its own disposal signal through, so `dispose()` aborts that wait immediately (synchronously, before its dispose callback runs) instead of hanging forever. Without this, `dispose()` had no bound at all against a peer that stops reading — the 2 second grace above only covers the flush-on-close path, not a write already blocked on backpressure.
- `@enkaku/transport`: `Transport.dispose()` now emits `disposed` even when the underlying stream never resolved (a failed connect). Previously, awaiting the writer sat outside the `try`/`catch`, so that rejection escaped the whole dispose callback — `disposed` never fired, and a `console.warn` fired instead. Patch bump: pure bug fix, no public API change, but restores an invariant (`disposed` always fires) that consumers may already depend on to release resources.
